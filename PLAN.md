# AI 정산 비서 기획서

> Agentic Workflow 기반 모임 정산 자동화 서비스  
> 참여자: 박상우 · 강형준 · 신재호 · 신채은 · 양성지

---

## 1. 프로젝트 개요

### 한 줄 정의

- **프로젝트**: 모임 정산 상황에서 참여자별 조건(술 미섭취, 늦은 도착, 선결제 등)을 자연어로 입력하면, LLM의 맥락 추론과 규칙 기반 계산 엔진을 조합하여 공정한 정산안과 그 근거를 함께 제공하는 Agentic Workflow 프로젝트
- **서비스**: 결제 금액과 상황만 알려주면, 공정한 기준으로 계산하고 왜 그렇게 나눴는지까지 설명해 주는 AI 정산 비서

### 배경 및 문제 정의

기존 정산 앱은 아래 3가지 문제를 해결하지 못한다.

1. **단순 N분의 1의 불공정함** — 술 미섭취자, 중도 귀가자의 상황이 반영되지 않음
2. **임의 감액의 설명 불가** — '왜 이 금액인가?'를 납득시키기 어려운 상황
3. **총무의 계산 피로** — 복잡한 조건을 사람이 직접 계산해야 하는 부담

### 대상 사용자

모임 후 총무 역할을 맡아 정산의 번거로움을 느끼는 20~30대 대학생 및 직장인

### 핵심 가치

- 정산 시간 단축: 복잡한 계산을 AI가 자동 처리
- 설명 가능한 결과: 각 참여자가 왜 해당 금액을 내는지 근거 제공
- 감정적 소모 최소화: 기준 있는 계산으로 불필요한 눈치 제거

## 2. Agent 설계

### Agent 역할

1. 자연어 입력에서 참여자·총 금액·비용 항목·예외 조건 추출
2. 입력 상황에 적합한 정산 전략 선택 (SIMPLE / EXCEPTION / SPONSOR)
3. 규칙 기반 엔진으로 참여자별 부담 금액 계산
4. 최소 부담 하한선 적용 및 총액 검증
5. 최종 정산표와 계산 근거 설명 제공
6. 피드백 입력 시 세션 내 재계산 수행

### Agent 톤 & 매너

- 중립적인 정산 조정자
- 특정 인물 비난 없이 "해당 항목 참여 여부를 기준으로 조정"하는 방식으로 설명
- 유능하고 투명한 회계사 톤

### Agent 자율성 범위

**수행하는 것:**

- 자연어 입력 분석 및 구조화 JSON 생성 (LLM)
- 정산 전략 분기 결정: SIMPLE / EXCEPTION / SPONSOR (LLM)
- 참여자별 정산 계산 및 하한선 적용 (규칙 엔진)
- 총액 검증 및 차액 보정
- 계산 근거 설명 생성 및 공유 메시지 포맷 출력
- 피드백 입력 시 세션 내 재계산

**수행하지 않는 것:**

- 실제 계좌 송금 또는 금융 트랜잭션 실행
- OCR, 카드 내역·계좌 연동
- 회원가입·로그인 및 장기 데이터 저장

## 3. 핵심 기능

### 기능 목록

| 기능                         | 담당                | 설명                                              |
| ---------------------------- | ------------------- | ------------------------------------------------- |
| ① 자연어 입력 분석 및 구조화 | LLM                 | 참여자, 총 금액, 비용 항목, 예외 조건 → JSON 추출 |
| ② 정산 전략 분기 결정        | LLM Decision Node   | SIMPLE / EXCEPTION / SPONSOR 분류                 |
| ③ 규칙 기반 정산 계산 엔진   | 규칙 엔진 (FastAPI) | N분의 1 기본 계산 + 예외 조건 조정 + 하한선 적용  |
| ④ 결과 설명 생성             | LLM                 | 금액 및 감액·제외 이유를 자연어로 설명            |
| ⑤ 피드백 기반 세션 내 재계산 | session_state       | 추가 조건만 반영해 재계산                         |

### 정산 전략 분기 상세

**SIMPLE**: 단순 N분의 1 분배

**EXCEPTION**: 아래 조건 중 하나 이상 존재 시

- 술 미섭취: 주류 비용에서 제외
- 음식 미섭취/소량 섭취: 안주 비용 일부 감액
- 늦은 도착: 참여 시간과 상황에 따라 일부 감액
- 중도 귀가: 이후 발생 비용에서 제외 또는 감액
- 특정 항목 미참여: 해당 항목 비용에서 제외

**SPONSOR**: 일부 인원 선결제 포함 복합 정산

### 계산 규칙

- 기본: 총액 ÷ 참여자 수
- 예외 조건 반영 후 조정
- **최소 부담 하한선: 균등 분담액의 30%** (과도한 불균형 방지)
- 총액 검증: 참여자별 합계 ↔ 입력 총액 비교
- 반올림 오차: 자동 보정 (차액 재분배)

## 4. 사용자 시나리오

### 시나리오 A — 술 미섭취 + 중도 귀가

- 4명(A, B, C, D), 총 8만원 (주류 3만 / 안주 5만)
- C는 늦게 도착, D는 술 미섭취
- 입력: `"총 8만원이고 A, B, C, D 있어. 주류 3만원 / 안주 5만원. C는 늦게 왔고 D는 술 안 마셨어."`
- 결과: 주류 비용 분리 + 예외 조건 반영한 정산표 제시

### 시나리오 B — 선결제 + 예외 조건 복합

- 5명(A~E), 총 120,000원 (주류 5만 / 안주 5만 / 공통비 2만)
- A가 50,000원 선결제, D는 술 미섭취, E는 늦게 도착해 안주 거의 미섭취
- 입력: `"총 12만원이고 A가 5만원 먼저 냈어. 주류 5만원, 안주 5만원, 공통비 2만원이야. D는 술 안 마셨고 E는 늦게 와서 안주를 거의 안 먹었어. 왜 이 금액인지 설명해줘."`
- 결과: 선결제 반영 + 항목별 참여 여부에 따른 정산 + 근거 설명

### 시나리오 C — 피드백 재계산

- 시나리오 B 이후, `"D는 안주도 거의 안 먹었어"` 추가 입력
- 기존 세션 정보 유지 + D의 안주 미섭취 조건만 반영해 재계산

## 5. 시스템 워크플로우

```
사용자 입력 (Streamlit UI)
         ↓
① InputParsingNode       : 자연어 → 구조화 JSON (Upstage LLM)
         ↓
② SafetyCheckNode        : 총액 누락·모순 감지 시 재입력 요청
         ↓
③ RouteRequestNode       : SIMPLE / EXCEPTION / SPONSOR 분기 (Upstage LLM)
         ↓
④ CalculationNode        : 규칙 기반 정산 엔진 실행 (FastAPI)
         ↓
⑤ FairnessAdjustNode     : 최소 부담 하한선 및 예외 조건별 보정 정책 적용
         ↓
⑥ ValidatorNode          : 총액 검증 및 차액 자동 보정
         ↓
⑦ ReportGenerationNode   : 계산 근거 설명 + 공유 메시지 생성 (Upstage LLM)
         ↓
Streamlit 화면 출력

[피드백 입력 시]
FeedbackParsingNode → 기존 세션 정보 + 피드백 비교
→ 수정 조건 구조화 → ④ CalculationNode 재진입
```

| 레이어             | 기술                 | 역할                                           |
| ------------------ | -------------------- | ---------------------------------------------- |
| Frontend / Demo UI | Streamlit            | 입력·결과 확인·피드백, session_state 세션 관리 |

- Streamlit 실행 주소: `http://localhost:8501`
- FastAPI 실행 주소: `http://localhost:8000`
- API Key 관리: `.env` 파일 또는 로컬 환경변수 (`UPSTAGE_API_KEY`)

## 7. 프롬프트 설계 전략

### InputParsingNode

- 역할: 자연어 → 참여자, 총 금액, 비용 항목, 예외 조건 JSON 추출
- 핵심 지시: **"금액 계산은 수행하지 말고, 구조화된 데이터 추출만 수행하라"**

### RouteRequestNode

- 역할: 입력 맥락 분석 → SIMPLE / EXCEPTION / SPONSOR 분류
- 핵심 지시: **"단어 기반이 아닌 문맥 기반으로 판단하라"**
  - 예: `"아 나 이따 가야해"` → 중도 귀가 조건으로 분류

### FeedbackParsingNode

- 역할: 피드백 → 기존 세션 데이터에 반영할 수정 조건으로 변환
- 입력: 기존 구조화 JSON + 이전 계산 결과 + 사용자 피드백 문장
- 출력: 추가·수정·삭제할 조건을 포함한 구조화 JSON
- 핵심 지시: **"기존 정산 정보를 임의로 덮어쓰지 말고, 사용자가 새로 말한 조건만 추가 또는 수정하라. 금액 계산은 수행하지 말고, 계산에 필요한 조건 변경 사항만 추출하라."**

### ReportGenerationNode

- 역할: 계산 결과 → 납득 가능한 자연어 설명 변환
- 핵심 지시: **"특정 참여자를 비난하지 말고, 항목 참여 여부를 기준으로 중립적으로 설명하라"**

## 9. 데이터 및 세션 관리

- **저장 방식**: Streamlit `session_state` (브라우저 종료 시 초기화 허용)
- **저장 대상**: 입력 원문, 구조화 JSON, 계산 결과, 피드백 이력
- **장기 저장 없음**: Vector DB, DB 연동 없음 (MVP 범위)
- **LangGraph State**: 노드 간 맥락 전달용

| 상황                          | 처리 방식                                |
| ----------------------------- | ---------------------------------------- |

| 참여자별 합계 ≠ 입력 총액     | 반올림 오차 보정 또는 차액 재분배        |
| 하한선 미달 케이스            | 균등 분담액의 30%를 최솟값으로 강제 적용 |

## 11. MVP 범위

### 반드시 구현

- Streamlit 기반 정산 입력·결과 확인 화면
- LangGraph StateGraph 기반 Agentic Workflow
- Upstage LLM 기반 자연어 입력 구조화 및 전략 분기 결정
- FastAPI 기반 정산 계산 API (SIMPLE / EXCEPTION / SPONSOR 3분기)
- 최소 부담 하한선(균등 분담액 30%) 및 총액 검증·차액 보정
- 참여자별 정산표 및 계산 근거 설명 출력
- `session_state` 기반 피드백 재계산
- 복사 가능한 공유용 정산 메시지 생성

### 구현 제외

- 실제 계좌 송금 및 금융 트랜잭션
- OCR, 카드 내역·계좌 연동
- 회원가입·로그인, 장기 데이터 저장

## 13. KPI (성공 지표)

- [ ] Streamlit 화면에서 자연어로 정산 상황 입력 가능
- [ ] Upstage LLM이 입력을 구조화된 JSON으로 변환
- [ ] LangGraph가 3가지 분기(SIMPLE / EXCEPTION / SPONSOR) 올바르게 라우팅
- [ ] FastAPI 정산 엔진이 참여자별 부담 금액 계산
- [ ] 술 미섭취·안주 미섭취·늦은 도착·선결제/지원금 조건 반영
- [ ] 최소 부담 하한선 적용으로 과도한 불균형 방지
- [ ] Settlement Validator가 총액 검증 및 차액 보정 수행
- [ ] 피드백 입력 시 세션 내 재계산 동작
- [ ] 3가지 대표 시나리오에서 전체 흐름(입력 구조화 → 계산 → 검증 → 설명) 시연 가능
